# 产品知识库构建通用标准

## 0. 知识库定位

产品知识库不是直接对外发布的话术库，也不是营销素材堆砌，而是用于沉淀产品事实、使用场景、适用边界、表达规范和证据依据的内部知识母库。

知识库的核心目标是：

1. 统一产品事实，避免后续内容引用错误。
2. 固定标准表达，降低不严谨话术被反复使用的风险。
3. 结构化沉淀核心场景、次相关场景和不适用场景。
4. 为后续生成对外文章、FAQ、AI 问答、GEO 内容、客服话术、销售物料提供可信依据。
5. 明确哪些内容有依据，哪些内容缺少依据，便于后续持续补证。

---

# 一、知识库整体结构标准

建议每个产品知识库都按照以下一级目录建设。

```markdown
# 产品知识库

## 0. 知识库使用说明
## 1. 产品总览
## 2. 产品身份信息
## 3. 官方/权威资料核心信息
## 4. 产品定位与核心价值
## 5. 核心适用场景
## 6. 次相关场景
## 7. 不适用/禁用/需谨慎场景
## 8. 作用机制与原理解释
## 9. 使用方法与操作规范
## 10. 产品差异与产品矩阵
## 11. 同类产品与竞品关系
## 12. 关联使用/联合使用知识边界
## 13. 标准表达库
## 14. 禁用表达库
## 15. 证据不足表达库
## 16. FAQ知识库
## 17. 合规与风险提示
## 18. 待核实信息与证据缺口
## 19. 版本记录与资料来源
```

> 注意：「用户自然语言表达映射」和「内容生成规则」已移出单产品知识库，作为产品线级全局模块独立管理，详见本标准附录一和附录二。

---

## 模块建设优先级

不是所有模块都必须同时完成。按优先级分为三级，避免因追求完整而导致知识库迟迟无法投入使用。

### P0 必建模块（缺少任一模块，知识库不可用）

| 模块 | 原因 |
|---|---|
| 产品身份信息 | 产品事实的基础，所有内容的根 |
| 官方/权威资料核心信息 | 最高优先级的依据源 |
| 核心适用场景 | 决定产品"能做什么" |
| 不适用/禁用/需谨慎场景 | 决定产品"不能做什么"，合规底线 |
| 标准表达库 | 统一对外口径，避免表述混乱 |
| 禁用表达库 | 拦截高风险表达，合规底线 |

### P1 应建模块（核心运营依赖，建议尽快补齐）

| 模块 | 原因 |
|---|---|
| 产品总览 | 快速理解产品全貌的入口 |
| 产品定位与核心价值 | 统一团队对产品的认知 |
| 次相关场景 | 明确产品的"灰色地带" |
| FAQ知识库 | 直接支撑对外内容生产 |
| 待核实信息与证据缺口 | 持续改进的驱动器 |

### P2 完善模块（逐步补充，持续丰富知识体系）

| 模块 | 原因 |
|---|---|
| 作用机制与原理解释 | 深度理解，非紧急 |
| 使用方法与操作规范 | 说明书已覆盖基础，细化可后置 |
| 产品差异与产品矩阵 | 需要跨产品信息，依赖其他产品知识库 |
| 同类产品与竞品关系 | 需要外部调研支撑 |
| 关联使用/联合使用知识边界 | 需要专业人员参与 |
| 证据不足表达库 | 在标准表达和禁用表达基础上逐步沉淀 |
| 合规与风险提示 | 可在场景模块中先行覆盖，独立整理可后置 |
| 版本记录与资料来源 | 建库即开始记录，但模块本身是被动的 |

---

## 单一事实源原则

同一个事实（如产品适应症、核心成分、禁忌条件等）在整个知识库中只允许有**一个权威维护位置**。其他模块如需使用该信息，应通过引用而非复制。

具体规则：

1. **产品总览**是摘要索引，不独立维护事实。其中的适用场景、风险限制等内容，应概括自对应模块，不展开细节。
2. **场景模块**中的"可使用表达"和"不建议表达"，应引用标准表达库和禁用表达库中的条目，不独立维护表达文本。
3. 当某条信息需要更新时，只需更新其权威维护位置，引用方自动获得最新版本。
4. 如果发现同一事实在多处出现且表述不一致，以事实的权威维护模块为准，其他位置应同步修正。

> 该原则的目标是：**改一处，全库一致；查一处，即为最新。**

# 二、知识库内容分层标准

产品知识库应分为五层。

## 1. 事实层

事实层用于记录产品最基础、最权威、最不可随意改写的信息。

包括：

* 产品名称
* 通用名称
* 商品名/品牌名
* 型号/规格
* 成分/材质/核心技术
* 剂型/形态/包装
* 批准文号/备案编号/认证信息
* 生产企业/上市许可持有人/制造商
* 适用范围/适应症/功能范围
* 使用方法
* 禁忌/限制条件
* 不良反应/副作用/风险事项
* 贮藏/保质期/有效期
* 官方说明书、检测报告、注册文件、专利、标准文件等来源

原则：

> 事实层优先使用官方文件、说明书、注册资料、检测报告、标准文件、监管信息，不应使用营销文案替代事实依据。

---

## 2. 解释层

解释层用于把专业事实转化为内部团队容易理解的知识，但不得改变事实边界。

包括：

* 专业术语解释
* 作用机制解释
* 使用逻辑解释
* 场景适配解释
* 产品差异解释
* 用户常见误解解释

示例格式：

```markdown
### 标准事实
用于某类明确场景。

### 知识库解释
该产品主要面向某类问题，可帮助解决某类需求，但不适用于所有相似问题。
```

原则：

> 解释可以通俗化，但不能扩大产品适用范围，不能弱化风险，不能把推测写成事实。

---

## 3. 场景层

场景层用于建立“用户需求—产品关系”的结构化映射。

应至少分为三类：

### 核心适用场景

指与官方资料、说明书、注册范围、产品设计用途高度一致的场景。

应记录：

* 场景名称
* 场景定义
* 用户常见表达
* 产品关联逻辑
* 可使用表达
* 不建议表达
* 风险提示
* 依据来源

### 次相关场景

指与产品功能或用户需求相关，但不属于最核心、最直接适用范围的场景。

应记录：

* 与产品的关联程度
* 是否可提及
* 是否可作为候选方案
* 是否需要条件限制
* 是否需要专业人员判断
* 不应承诺的内容

### 不适用/禁用/需谨慎场景

指存在明显不匹配、风险较高、需要专业判断或可能误用的场景。

应记录：

* 不适用原因
* 可能风险
* 正确处理建议
* 是否需要转向专业咨询
* 禁止表达

原则：

> 场景层不是为了无限扩大产品覆盖范围，而是为了明确“什么时候可以推荐、什么时候可以提及、什么时候不能推荐”。

---

## 4. 表达层

表达层用于固定标准说法，减少后续内容生产中的自由发挥。

包括：

* 标准事实表达
* 标准作用表达
* 标准场景表达
* 标准使用表达
* 标准风险表达
* 标准对比表达
* 禁用表达
* 证据不足表达

原则：

> 能固定的话术尽量固定，尤其是涉及功效、适用范围、风险、安全性、对比优势、特殊人群、长期使用等内容。

---

## 5. 证据层

证据层用于标注每类信息的依据质量，防止无依据表达被当作事实。

每条重要信息建议标注证据等级。

| 证据等级 | 定义                      | 可用方式          |
| ---- | ----------------------- | ------------- |
| A级   | 官方说明书、注册资料、监管文件、国家/行业标准 | 可作为核心事实       |
| B级   | 临床研究、系统综述、指南、专家共识、检测报告  | 可作为专业支撑       |
| C级   | 企业内部资料、培训资料、产品手册、客服经验   | 可作为内部参考       |
| D级   | 市场反馈、用户评价、销售经验、内容观察     | 仅作启发，不宜直接作为事实 |
| E级   | 暂无来源、待核实、推测性表达          | 不应直接对外使用      |

### 混合证据的标注规则

实际中，一条知识往往涉及多个证据等级。例如"糠酸莫米松凝胶用于湿疹"来自说明书（A级），但"湿疹急性渗出期优先选凝胶剂型"可能来自临床经验（C/D级）。

处理原则：**不混标，拆开标。**

当一条表述涉及不同证据等级的判断时，应将其拆分为独立表述，分别标注证据等级。这样做的好处是：

1. 内部人员可以清楚看到哪部分是最强依据、哪部分是最薄弱环节。
2. 后续补证时，可以针对低等级部分精准补充，而非整条推翻重写。
3. 对外使用时，可以只使用高证据等级部分，低等级部分暂不输出。

示例：

```markdown
### 拆分前（不推荐）
糠酸莫米松凝胶适用于湿疹，急性渗出期优先选择凝胶剂型。【证据等级：？】

### 拆分后（推荐）
- 糠酸莫米松凝胶适用于湿疹。【证据等级：A级 | 来源：药品说明书】
- 湿疹急性渗出期，凝胶剂型因质地清爽、不封闭渗出面，可能更适配。【证据等级：C级 | 来源：临床用药经验】
```

# 三、每个模块的具体建设标准

## 1. 产品总览

用于快速说明产品是什么、解决什么问题、适合谁、不适合谁。

建议字段：

```markdown
## 产品总览

- 产品类别：
- 产品核心功能：
- 主要适用人群/场景：
- 核心价值：
- 主要使用方式：
- 主要风险/限制：
- 与同类产品的主要区别：
- 资料来源：
```

写法要求：

* 简洁、客观。
* 不使用夸张营销词。
* 不承诺超出资料来源的结果。

---

## 2. 产品身份信息

用于沉淀产品基础事实。

建议字段：

```markdown
## 产品身份信息

- 产品名称：
- 通用名称：
- 商品名/品牌名：
- 英文名称：
- 型号/规格：
- 成分/核心材料/核心技术：
- 产品形态：
- 批准文号/备案编号/认证编号：
- 执行标准：
- 生产企业/制造商：
- 上市许可持有人/责任主体：
- 包装规格：
- 贮藏/保存条件：
- 有效期/保质期：
- 资料来源：
```

要求：

* 所有编号、企业名称、规格必须逐字核对。
* 不确定信息放入“待核实信息与证据缺口”，不要写进核心事实区。

---

## 3. 官方/权威资料核心信息

该模块应尽量保留原始权威资料的核心表达。

建议分为：

```markdown
### 官方标准表达
保留原文或接近原文。

### 知识库解释表达
在不改变原意的情况下进行通俗解释。

### 可用于对外内容的表达
转化为更适合文章、FAQ、AI回答的稳妥表达。

### 不可改写边界
列出不能扩大的范围、不能弱化的风险。
```

示例：

```markdown
### 官方标准表达
适用于A、B、C场景。

### 知识库解释表达
该产品主要用于A、B、C相关问题，不代表所有类似症状均适用。

### 不可改写边界
不得改写为“所有相关问题都可以使用”。
```

---

## 4. 产品定位与核心价值

该模块用于形成产品在知识体系中的位置。

建议字段：

```markdown
## 产品定位

- 产品类别定位：
- 目标问题：
- 核心作用：
- 核心使用场景：
- 主要使用人群：
- 主要差异点：
- 不应承诺的内容：
```

允许表达：

* “用于……”
* “适用于……”
* “可帮助……”
* “主要面向……”
* “在……场景下可作为一种选择”

慎用表达：

* “最佳”
* “最安全”
* “最快”
* “根治”
* “彻底解决”
* “无副作用”
* “所有人都适合”

---

## 5. 核心适用场景

每个核心场景应独立成节。

模板：

```markdown
## 场景：XXX

### 场景等级
核心适用场景。

### 场景定义
解释该场景是什么。

### 用户常见表达
- 表达1
- 表达2
- 表达3

### 产品关联逻辑
说明为什么该产品与该场景相关。

### 关联表达（引用标准表达库和禁用表达库）
- 可使用：引用标准表达库条目编号，如 [SE-001]、[SE-002]
- 禁止使用：引用禁用表达库条目编号，如 [FE-001]、[FE-002]

> 场景模块不独立维护表达文本。所有标准表达和禁用表达统一在表达库中管理，场景模块仅引用。

### 风险提示
说明哪些情况下不能直接使用或需要专业判断。

### 依据来源
列出说明书、注册资料、指南、研究、检测报告等。
```

核心场景的表达强度可以较高，但仍应基于依据。

推荐使用：

* “适用于”
* “可用于”
* “可作为……选择”
* “符合……场景时可考虑”

避免使用：

* “一定有效”
* “适合所有”
* “无需判断”
* “长期使用也没问题”

---

## 6. 次相关场景

模板：

```markdown
## 场景：XXX

### 场景等级
次相关场景。

### 与产品的关系
说明该场景与产品功能、使用需求或用户问题的关联。

### 可提及逻辑
说明在什么条件下可以提及产品。

### 不应直接推荐的原因
说明为什么不能把该场景直接归入核心适用范围。

### 关联表达（引用标准表达库和禁用表达库）
- 可使用：引用标准表达库条目编号
- 禁止使用：引用禁用表达库条目编号

> 场景模块不独立维护表达文本，统一引用表达库。

### 风险提示
列出需要排除的情况。
```

原则：

> 次相关场景的主要价值是“建立产品关联”，不是“强推荐”。

---

## 7. 不适用/禁用/需谨慎场景

建议分为三类。

### 禁用场景

官方明确禁止或不适合的场景。

表达方式：

```markdown
该场景不应推荐使用本产品。若用户出现该情况，应建议停止自行使用并咨询专业人员。
```

### 不建议自行使用场景

信息不足、风险较高或容易误用的场景。

表达方式：

```markdown
该场景不建议用户自行判断使用，应结合具体情况并咨询专业人员。
```

### 需谨慎场景

特殊人群、特殊部位、长期使用、高频使用、联合使用等。

表达方式：

```markdown
该场景下需谨慎使用，建议在专业人员指导下判断是否适合。
```

模板：

```markdown
## 不适用/禁用/需谨慎场景：XXX

### 场景类型
禁用 / 不建议自行使用 / 需谨慎。

### 原因
说明风险原因。

### 标准处理建议
说明应如何引导。

### 禁止表达
- 可放心使用
- 没问题
- 直接用即可
```

---

# 四、标准表达库建设规则

标准表达库是整个知识库最重要的模块之一。

## 0. 表达库管理原则

**标准表达库和禁用表达库是产品知识库中所有表达的唯一权威源。**

其他模块（如场景模块、FAQ模块）中涉及的表达，应引用表达库中的条目编号，而非独立编写表达文本。这样做的目的是：

1. 修改一条表达时，只需在表达库中更新一处，所有引用方自动同步。
2. 避免同一产品的不同模块中出现措辞不一致的情况。
3. 便于后续对表达进行批量审计和合规检查。

### 表达条目编号规则

每条标准表达和禁用表达应分配唯一编号，便于引用和追溯。

```markdown
标准表达编号：SE-{产品缩写}-{序号}，如 SE-PYP-001（皮炎平标准表达第1条）
禁用表达编号：FE-{产品缩写}-{序号}，如 FE-PYP-001（皮炎平禁用表达第1条）
```

表达库条目格式：

```markdown
| 编号 | 表达类型 | 表达内容 | 适用场景 | 证据等级 | 依据来源 |
|---|---|---|---|---|---|
| SE-PYP-001 | 事实表达 | 999皮炎平（糠酸莫米松凝胶）是一种外用糖皮质激素类药物，适用于湿疹、神经性皮炎、异位性皮炎及皮肤瘙痒症。 | 通用 | A级 | 药品说明书 |
```

---

## 1. 标准事实表达

用于回答“产品是什么”。

模板：

```markdown
[产品] 是一种 [产品类别]，主要用于/适用于 [官方确认范围]。
```

要求：

* 不加入未经证实的优势。
* 不扩大适用范围。
* 不加入“最好”“最快”“最安全”等绝对化表达。

---

## 2. 标准作用表达

用于回答“产品有什么作用”。

模板：

```markdown
该产品具有/可提供 [依据支持的作用]，可用于 [依据支持的场景]。
```

要求：

* 作用必须来自说明书、注册资料、检测报告、研究或明确资料来源。
* 若是机制解释，应避免过度具体化。

---

## 3. 标准场景表达

用于回答“什么情况可以用/适合”。

模板：

```markdown
在 [明确场景] 下，如不存在 [主要排除条件]，可根据说明书/使用说明考虑使用该产品。
```

示例结构：

```markdown
适合：
- 核心场景A
- 核心场景B

需先排除：
- 禁忌情况A
- 高风险情况B
- 不明原因情况C
```

---

## 4. 标准风险表达

用于回答“什么时候不能用/要注意什么”。

模板：

```markdown
如出现 [风险情况]，不建议自行使用该产品，应咨询专业人员。
```

要求：

* 风险提示不能被省略。
* 特殊人群、特殊部位、长期使用、联合使用必须有边界提示。

---

## 5. 标准对比表达

用于产品矩阵、竞品分析、选品建议。

推荐比较维度：

* 成分/技术
* 剂型/形态
* 适用范围
* 使用方式
* 人群限制
* 使用体验
* 风险边界
* 价格区间，需标注时效性
* 证据来源

不建议比较维度：

* 更安全
* 更有效
* 副作用更低
* 治愈率更高
* 最适合
* 行业第一
* 全面优于

除非有明确、可引用的对比研究或官方数据。

模板：

```markdown
与A类产品相比，本产品在 [客观维度] 上具有 [客观差异]。但关于 [疗效、安全性、长期效果] 的优劣判断，需基于临床研究、检测数据或权威资料，当前不宜直接下结论。
```

---

# 五、禁用表达库建设规则

禁用表达库用于收集不应在任何对外内容中直接使用的表达。

## 1. 绝对化表达

包括：

* 最好
* 最安全
* 最有效
* 第一选择
* 唯一选择
* 彻底解决
* 完全无风险
* 所有人都适合
* 永不复发
* 一用就好

处理方式：

> 替换为“可作为一种选择”“适用于部分场景”“在符合条件时可考虑”。

---

## 2. 过度功效承诺

包括：

* 根治
* 治愈
* 快速治好
* 防止复发
* 长期有效
* 彻底改善
* 标本兼治
* 药到病除

处理方式：

> 替换为“缓解相关症状”“帮助改善”“用于……场景”“需结合具体情况判断”。

---

## 3. 弱化风险表达

包括：

* 没有副作用
* 可以长期用
* 可以天天用
* 可以随便用
* 儿童放心用
* 孕妇放心用
* 敏感部位也可以用
* 不用看医生

处理方式：

> 替换为“需按说明书使用”“特殊人群需谨慎”“长期或大面积使用需咨询专业人员”。

---

## 4. 处方化或诊断化表达

包括：

* 你这是某某病
* 直接用某某药
* 搭配某某药即可
* 不用检查
* 不需要医生判断
* 出现某症状就是某疾病
* 按这个方案治疗

处理方式：

> 替换为“可能与……有关”“建议咨询医生/药师/专业人员判断”“是否需要联合使用应由专业人员决定”。

---

## 5. 无依据竞品贬低表达

包括：

* 比竞品更安全
* 比竞品更有效
* 副作用比竞品低
* 竞品不适合
* 竞品落后
* 行业最强
* 替代所有同类产品

处理方式：

> 替换为基于成分、技术、形态、适用场景、使用体验等客观差异的描述。

---

# 六、证据不足表达库建设标准

当前缺少依据、暂不能作为标准话术的内容，作为后续补充临床证据、检测数据、用户研究、专家共识、监管资料的任务池。

## 1. 证据不足表达的定义

以下表达应纳入“证据不足表达库”：

1. 当前缺少临床研究支持的功效判断。
2. 当前缺少安全性数据支持的安全判断。
3. 当前缺少同类对比研究支持的竞品优劣判断。
4. 当前缺少长期随访或真实世界数据支持的长期效果判断。
5. 当前缺少特殊人群研究支持的人群适用判断。
6. 当前缺少价格、医保、政策、渠道来源的时效性信息。
7. 当前仅来自销售经验、用户反馈、培训材料，但未找到外部证据的表达。
8. 当前表述过于确定，但依据只支持较弱结论的表达。

---

## 2. 证据不足表达库字段

建议使用表格管理。

```markdown
| 原始表达 | 表达类型 | 当前问题 | 所需证据 | 当前处理方式 | 后续补证方向 |
|---|---|---|---|---|---|
|  | 功效/安全/竞品/人群/价格/机制/长期效果 | 缺少什么依据 | 需要什么资料支持 | 暂不对外使用/降级表达/仅内部参考 | 后续需要补充什么 |
```

---

## 3. 常见证据不足表达类型

### 类型一：功效速度类

常见表达：

* 快速起效
* 快速止痒
* 很快见效
* 一涂就缓解
* 短时间内明显改善

当前问题：

> 缺少起效时间、有效率、症状评分变化等临床数据支持。

所需证据：

* 临床研究
* 用户研究
* 真实世界数据
* 症状评分变化数据
* 起效时间统计

当前处理方式：

> 可降级为“可帮助缓解相关症状”，不直接承诺起效速度。

---

### 类型二：疗效强度类

常见表达：

* 效果更好
* 疗效更强
* 明显优于同类产品
* 治疗效果突出
* 对顽固问题也有效

当前问题：

> 缺少与同类产品或安慰剂对照的临床数据支持。

所需证据：

* 头对头临床研究
* 指南推荐
* Meta 分析
* 真实世界研究
* 适用人群分层数据

当前处理方式：

> 可保留“适用于某些场景”，不表达“更有效”。

---

### 类型三：安全性优越类

常见表达：

* 更安全
* 副作用更低
* 安全性更高
* 可以放心用
* 长期用也安全
* 儿童也安全
* 老人也安全

当前问题：

> 缺少特定人群、长期使用、不良反应发生率、同类对比安全性数据。

所需证据：

* 安全性研究
* 不良反应发生率数据
* 特殊人群研究
* 长期随访数据
* 药品警戒数据
* 监管资料

当前处理方式：

> 降级为“按说明书规范使用，并注意特殊人群和长期使用风险”。

---

### 类型四：适用人群扩大类

常见表达：

* 儿童可用
* 孕妇可用
* 老人可用
* 敏感肌可用
* 全家可用
* 所有人都适合
* 娇嫩肌肤也适合

当前问题：

> 可能扩大官方适用范围，且缺少特殊人群数据或限制条件。

所需证据：

* 官方说明书
* 特殊人群研究
* 专家共识
* 儿童/孕妇/老人使用数据
* 监管批准信息

当前处理方式：

> 改为“特殊人群需谨慎，建议咨询专业人员”。

---

### 类型五：长期效果类

常见表达：

* 减少复发
* 防止复发
* 长期改善
* 调理根源
* 修复屏障后不再反复
* 标本兼顾

当前问题：

> 缺少长期随访、复发率、停用后效果维持等数据。

所需证据：

* 长期随访研究
* 复发率统计
* 真实世界数据
* 停用后观察数据

当前处理方式：

> 可表达为“帮助缓解当前症状”，不承诺复发控制。

---

### 类型六：机制过度具体类

常见表达：

* 直接阻断某某通路
* 精准作用于某某受体
* 显著降低某某炎症因子
* 提高生物利用度
* 深层修复
* 从根源调节
* 激活细胞再生

当前问题：

> 机制描述过于具体，缺少产品相关研究或实验数据支持。

所需证据：

* 药理研究
* 机制研究
* 体外实验
* 动物实验
* 人体研究
* 产品专属技术资料

当前处理方式：

> 降级为官方资料支持的机制概括表达。

---

### 类型七：剂型/技术优势类

常见表达：

* 吸收更快
* 渗透更深
* 利用率更高
* 效果更持久
* 不刺激
* 更温和
* 更适合敏感人群

当前问题：

> 缺少透皮吸收、刺激性测试、稳定性测试、用户体验研究或同类对比数据。

所需证据：

* 透皮吸收研究
* 刺激性测试
* 稳定性测试
* 使用体验调研
* 同类剂型对比研究

当前处理方式：

> 可保留客观剂型描述，如“凝胶质地相对清爽”“乳膏质地相对滋润”，不直接推导为疗效或安全优势。

---

### 类型八：价格/医保/渠道类

常见表达：

* 价格十几元
* 能走医保
* 药店都能买到
* 网上随时可以买
* 性价比最高
* 更划算

当前问题：

> 价格、医保、渠道信息具有时效性和地区差异，缺少最新来源。

所需证据：

* 最新电商价格
* 药店价格调研
* 医保目录
* 地方医保政策
* 渠道销售资料

当前处理方式：

> 改为“价格、医保和购买渠道以当地政策及实际销售渠道为准”。

---

### 类型九：用户体验泛化类

常见表达：

* 大多数人都说好
* 用户反馈很好
* 不油腻
* 不粘
* 用起来舒服
* 口碑很好

当前问题：

> 用户体验表述可能来自个体反馈，缺少样本量、调研方法、统计数据。

所需证据：

* 用户调研
* 样本量说明
* 问卷数据
* 真实世界反馈分析
* 第三方口碑数据

当前处理方式：

> 可表达为“质地特点为……”，不要泛化为所有用户体验。

---

### 类型十：竞品优劣类

常见表达：

* 比A更好
* 比B更安全
* 比C更温和
* 比D更适合儿童
* 替代某某产品
* 优于传统方案

当前问题：

> 缺少直接对比研究或权威评价。

所需证据：

* 头对头研究
* 第三方检测
* 指南推荐
* 专家共识
* 真实世界对比数据

当前处理方式：

> 仅保留客观差异，不做优劣判断。

---

## 4. 证据不足表达的处理等级

建议分为四级处理。

| 处理等级 | 说明              | 使用方式          |
| ---- | --------------- | ------------- |
| 保留待证 | 表达可能有价值，但当前缺少来源 | 放入证据不足库，不对外使用 |
| 降级使用 | 原表达过强，但可改成稳妥表达  | 用于内部和对外内容     |
| 内部参考 | 来自经验或观察，可辅助理解   | 不作为标准话术       |
| 禁止使用 | 明显违规、误导或高风险     | 不得用于任何输出      |

示例：

```markdown
| 原始表达 | 当前处理方式 | 替代表达 |
|---|---|---|
| 快速止痒 | 降级使用 | 可帮助缓解瘙痒症状 |
| 儿童放心用 | 禁止使用 | 儿童使用需谨慎，建议咨询专业人员 |
| 比竞品更安全 | 保留待证 | 与同类产品相比，在剂型和适用场景上存在差异 |
```

# 七、（已移至附录一）用户自然语言表达映射标准

> **重要变更**：用户自然语言表达映射不再放在单产品知识库中，而是作为产品线级全局模块独立管理。原因见下方说明，完整标准见本文档附录一。

**为什么不放在单产品内：**

用户的自然语言搜索（如"皮肤痒用什么药""孩子感冒怎么办"）天然是跨产品的，不针对某一个具体产品。如果每个产品各自维护一份用户表达映射：

1. 同一条用户表达会在多个产品知识库中重复出现，维护成本高。
2. 无法实现跨产品的意图匹配和推荐排序。
3. 各产品的映射之间可能出现矛盾（A产品说该场景"推荐"，B产品也说"推荐"，但两者不应同时推荐）。

正确做法是建立产品线级别的全局映射表，结构为：

```
用户表达 → 意图识别 → 关联产品列表（含各产品的关联程度和推荐动作）
```

---

# 八、关联使用/联合使用知识边界

很多产品会与其他产品、服务或方案共同出现在用户场景中。知识库可以记录这种关联，但不能把它直接写成无条件推荐方案。

建议字段：

```markdown
| 场景 | 可能关联的产品/服务类型 | 本产品角色 | 是否可直接推荐组合 | 风险边界 | 依据来源 |
|---|---|---|---|---|---|
```

表达原则：

* 可以写“常见共现场景”。
* 可以写“可能同时被讨论”。
* 可以写“是否需要联合使用应由专业人员判断”。
* 不建议写“固定搭配方案”。
* 不建议写“出现某症状就搭配某产品”。

标准表达：

```markdown
在该场景中，用户可能同时关注A类产品、B类产品和C类产品。本产品可作为其中的相关选项之一。是否需要联合使用，应结合具体情况并由专业人员判断。
```

---

# 九、FAQ知识库建设标准

FAQ 应分为内部知识问答和对外参考问答。

## 1. 内部知识问答

用于团队理解产品，不直接对外。

可包含：

* 为什么适用于该场景？
* 为什么不适用于该场景？
* 和同类产品有什么区别？
* 用户常见误解是什么？
* 哪些表达不能用？
* 当前缺少哪些证据？

## 2. 对外参考问答

用于后续生成文章、客服问答、GEO内容，但仍需经过渠道适配。

格式：

```markdown
Q：
A：
依据来源：
风险提示：
可改写范围：
不可改写边界：
```

原则：

> FAQ 不是越口语越好，而是要保证每个答案都可追溯、可复用、可控风险。

# 十、内容生成规则

> **该模块已移至附录二，作为产品线级全局规范独立管理。** 单产品知识库中不再重复放置内容生成规则。所有产品在生成对外内容时，统一遵守附录二中的规则。

---

# 十一、待核实信息与证据缺口标准

每个知识库都应保留一个持续更新模块。

模板：

```markdown
## 待核实信息与证据缺口

| 信息点 | 当前状态 | 缺口类型 | 需要补充的资料 | 优先级 | 负责人 | 更新时间 |
|---|---|---|---|---|---|---|
```

缺口类型包括：

* 官方资料缺口
* 临床证据缺口
* 安全性数据缺口
* 特殊人群数据缺口
* 竞品对比数据缺口
* 价格/医保/渠道时效性缺口
* 用户体验数据缺口
* 机制研究缺口
* 长期效果数据缺口

优先级建议：

| 优先级 | 定义                     |
| --- | ---------------------- |
| P0  | 涉及合规、安全、禁忌、适用范围，必须优先核实 |
| P1  | 涉及核心卖点、核心场景、对外高频表达     |
| P2  | 涉及内容丰富度、科普解释、竞品理解      |
| P3  | 涉及市场表达、用户体验、低频问题       |

---

# 十二、知识库版本管理标准

每次更新应记录：

```markdown
## 版本记录

| 版本 | 更新时间 | 更新内容 | 更新原因 | 资料来源 | 更新人 |
|---|---|---|---|---|---|
```

要求：

* 所有新增内容必须记录来源。
* 所有删除或降级表达必须记录原因。
* 所有证据不足内容必须进入证据不足表达库或待核实模块。
* 对外高频表达变更，应单独标记。

---

# 十三、最终判断规则

构建产品知识库时，可以用以下问题检查质量：

1. 这条信息有没有来源？
2. 来源等级是什么？
3. 表达强度是否超过证据强度？
4. 是否扩大了产品适用范围？
5. 是否弱化了风险或禁忌？
6. 是否把经验判断写成事实？
7. 是否存在绝对化、治疗化、处方化、诊断化表达？
8. 是否标注了不适用场景？
9. 是否给后续内容生产提供了标准表达？
10. 是否把缺证据但有价值的表达沉淀到了证据不足库？
11. 同一事实是否只在一个模块中维护？是否存在多处重复？
12. 场景模块中的表达是否引用了表达库，而非独立编写？
13. 涉及混合证据的表述，是否已拆分并分段标注证据等级？

# 附录一：用户自然语言表达映射标准（产品线级全局模块）

该模块独立于单产品知识库之外，由产品线统一维护，用于连接用户真实问题和产品知识。

## 1. 为什么是全局模块

用户搜索"皮肤痒用什么""孩子发烧怎么办"等自然语言，天然是跨产品的。全局映射表可以：

1. 在一处统一管理所有用户表达，避免各产品重复收录。
2. 支持跨产品的意图匹配和推荐优先级排序。
3. 避免多个产品对同一用户表达给出矛盾的推荐动作。

## 2. 模板

```markdown
| 用户表达 | 可能意图 | 关联产品 | 对应场景 | 关联程度 | 推荐动作 | 风险排查 |
|---|---|---|---|---|---|---|
| 皮肤痒用什么药 | 寻求止痒方案 | 皮炎平绿皮 | 核心适用 | 高 | 条件推荐 | 需排除真菌感染、严重过敏等 |
| 皮肤痒用什么药 | 寻求止痒方案 | 皮炎平红皮 | 核心适用 | 高 | 条件推荐 | 需排除真菌感染、严重过敏等 |
| 孩子感冒发烧 | 儿童感冒对症 | 小儿氨酚黄那敏 | 核心适用 | 高 | 条件推荐 | 需确认年龄、体重、排除重复用药 |
```

## 3. 推荐动作定义

* **推荐**：核心场景且风险较低。
* **条件推荐**：核心或强相关场景，但需排除风险因素。
* **提及**：次相关场景，仅作为候选项或相关知识点。
* **不推荐**：不适用或高风险场景。
* **建议专业咨询**：涉及特殊人群、严重症状、长期问题、不明原因等。

## 4. 原则

> 用户自然语言映射不是为了扩大推荐范围，而是为了识别用户真实意图，并把回答导向正确的产品和知识模块。当同一用户表达关联多个产品时，应根据场景匹配程度和产品关联程度排序，而非让每个产品都抢"推荐"位。

# 附录二：内容生成规则（产品线级全局规范）

该规则对所有产品通用，独立于单产品知识库之外，由产品线统一维护。各产品知识库在生成对外内容时，必须遵守本规范。

## 1. 资料优先级规则

当不同资料之间有冲突时，优先级如下：

1. 法规、监管文件、注册资料、说明书、官方标准。
2. 临床研究、指南、专家共识、检测报告。
3. 企业正式产品资料。
4. 内部培训资料。
5. 用户反馈、销售经验、市场观察。
6. 未核实表达。

> 高优先级资料与低优先级资料冲突时，以高优先级资料为准。

## 2. 表达强度规则

表达强度应与证据强度匹配。

| 证据强度 | 可用表达            |
| ---- | --------------- |
| 强证据  | 适用于、可用于、具有……作用  |
| 中等证据 | 可帮助、可能有助于、可作为选择 |
| 弱证据  | 可作为内部参考、可能相关    |
| 无证据  | 待核实，不建议对外使用     |

## 3. 风险提示规则

以下内容必须配套风险提示：

* 功效相关内容
* 使用方法
* 特殊人群
* 长期使用
* 大面积使用
* 联合使用
* 敏感部位
* 禁忌场景
* 不良反应
* 与同类产品比较
* 价格、医保、渠道信息

## 4. 对外内容改写规则

对外内容可以更通俗，但不得：

* 改变适用范围
* 扩大适用人群
* 弱化禁忌风险
* 承诺疗效结果
* 引导高风险自用
* 贬低竞品
* 使用无依据绝对化表达
* 把内部经验写成事实
